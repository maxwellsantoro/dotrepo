# Public messaging notes

## Desired tone

Public communications should balance:
- excitement about what becomes possible
- idealism about the shared infrastructure value
- seriousness about trust, provenance, and respecting source materials
- practical clarity about what dotrepo does and does not claim

## Framing guidance

### Good framing
- open metadata protocol
- reference toolchain
- public index
- machine-readable repository layer
- trust-aware overlays
- practical compatibility with existing repository surfaces

### Framing to avoid
- replacing documentation with AI summaries
- eliminating README culture
- scraping and rewriting projects without accountability
- claiming more certainty than the provenance supports
- treating repositories as raw inputs to harvest without respect

## Suggested public blurb

**dotrepo** is an open metadata protocol for software repositories, with a reference toolchain and a public index. It helps maintainers, users, and coding agents share the same structured understanding of a project by making essential repository metadata machine-readable, queryable, and trust-aware. dotrepo supports canonical in-repo records as well as carefully labeled overlays for public repositories, so projects can become mechanically visible without losing provenance, nuance, or respect for their original materials.

## Trust-building points to repeat

- dotrepo distinguishes canonical, imported, inferred, reviewed, and verified records
- overlays should show how claims were derived, not just what they claim
- source materials remain primary; dotrepo is a metadata layer, not a replacement
- maintainers can claim and correct how their project is represented
