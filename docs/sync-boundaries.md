# Sync boundaries and non-round-trippable cases

This page describes the current managed-sync boundary for dotrepo. It is the
practical companion to [`RFC 0005`](../rfcs/0005-sync-and-generated-artifacts.md).

The most important rule is unchanged:

**The root `.repo` file is the only source of truth.** Managed sync exists to
keep selected conventional files aligned with `.repo` without pretending that
dotrepo can safely round-trip arbitrary prose.

## Supported surfaces

Managed-region sync currently applies only to these Markdown surfaces:

- `README.md`
- `.github/SECURITY.md` or `SECURITY.md`
- `CONTRIBUTING.md` or `.github/CONTRIBUTING.md`

The current managed-region ids are:

- `readme.body`
- `security.body`
- `contributing.body`

Other conventional files such as `CODEOWNERS` or pull request templates can
still be fully generated by dotrepo, but they do not support partial managed
regions.

## State model

For supported Markdown surfaces, dotrepo currently uses these states:

| State | Meaning | `doctor` | `generate --check` | `generate` |
| --- | --- | --- | --- | --- |
| `fully_generated` | The whole file is dotrepo-owned and identified by the generated banner. | Reported explicitly. | Fails if the full file drifts. | Rewrites the full file. |
| `partially_managed` | The file contains valid managed-region markers for the required surface id. | Reported explicitly. | Fails only if the managed region drifts. | Rewrites only the managed region and preserves content outside it. |
| `unmanaged` | The file exists without a dotrepo banner or managed markers. | Reported explicitly. | Does not fail solely because the file exists. | Leaves the file alone. |
| `malformed_managed` | Markers are present but broken or ambiguous. | Reported explicitly with an actionable message. | Fails. | Refuses to rewrite. |
| `unsupported` | The file layout is outside the supported managed-sync contract. | Reported explicitly with an actionable message. | Fails when the unsupported state affects a managed surface. | Refuses to guess. |

`generate --check` may also surface `missing` in machine-readable output when a
managed or fully generated surface is expected but absent on disk. `doctor`
does not list missing files because there is no on-disk surface to inspect.

## What dotrepo preserves

dotrepo preserves:

- unmanaged prose outside valid managed markers in supported Markdown files
- the full contents of explicitly unmanaged files
- fully generated file banners and digest metadata
- stable managed-region ids for the currently supported surfaces

dotrepo does not preserve or infer authority from edits inside managed regions.
Those edits are treated as drift against `.repo`.

## What dotrepo will not attempt

dotrepo currently does **not** attempt:

- arbitrary prose merging outside the supported managed-region surfaces
- automatic conversion of an unmanaged file into a managed-region file
- partial management for `CODEOWNERS`
- partial management for pull request templates
- nested or overlapping managed regions
- multiple candidate files for the same managed surface at once
- silent fallback from a broken or unsupported managed layout into a guessed one

## Concrete examples

### 1. Fully generated README

If `README.md` begins with the dotrepo generated banner, dotrepo treats it as
`fully_generated`.

- `doctor` reports `README.md [fully_generated]`
- `generate --check` fails if any part of the file drifts
- `generate` rewrites the whole file from `.repo`

### 2. Partially managed README with local prose around it

If `README.md` contains:

```html
<!-- dotrepo:begin id=readme.body -->
...
<!-- dotrepo:end id=readme.body -->
```

then dotrepo treats it as `partially_managed`.

- prose before and after the managed region is preserved
- only the managed block is rewritten
- `generate --check` compares only the managed block, not the surrounding prose

### 3. Unmanaged README with no markers

If `README.md` exists with no banner and no managed markers, dotrepo treats it
as `unmanaged`.

- `doctor` reports `README.md [unmanaged]`
- `generate --check` does not fail solely because it exists
- `generate` does not overwrite it implicitly

This is the intentional guardrail that prevents dotrepo from silently replacing
hand-written prose.

### 4. Malformed managed SECURITY file

If `.github/SECURITY.md` opens a marker but never closes it, or closes the wrong
id, dotrepo treats it as `malformed_managed`.

- `doctor` reports the malformed state and includes the marker diagnostic
- `generate --check` fails
- `generate` refuses to rewrite until the markers are fixed

### 5. Unsupported managed layout

If both `SECURITY.md` and `.github/SECURITY.md` exist at the same time, or if a
managed file uses the wrong region id for that surface, dotrepo treats the
surface as `unsupported`.

- `doctor` reports the unsupported state with an explanation
- `generate --check` fails for the managed surface
- `generate` refuses to guess which file or region is authoritative

### 6. Unsupported partial management for CODEOWNERS

`CODEOWNERS` can be fully generated, but it does not support managed-region
markers.

- a generated `.github/CODEOWNERS` file is fine as a full-surface output
- a hand-written `CODEOWNERS` file remains outside the managed-region contract
- `doctor` reports that unsupported state instead of pretending partial sync
  exists there

## Practical guidance

- Use `fully_generated` when dotrepo should own the whole file.
- Use `partially_managed` only on the supported Markdown surfaces and only with
  the correct region id.
- Leave a file `unmanaged` if you are not ready to hand control of that region
  to dotrepo.
- Treat `malformed_managed` and `unsupported` as repair states, not as soft
  warnings.

If you need a surface that is not covered here, the current answer is to leave
it unmanaged or fully generated. dotrepo does not yet support a richer
round-tripping contract than that.
